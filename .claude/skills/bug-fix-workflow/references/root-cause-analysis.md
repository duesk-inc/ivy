# 根本原因分析（Root Cause Analysis）

## 概要

表面的な症状ではなく、本質的な原因を特定するための体系的分析手法。
「なぜ？」を繰り返し問うことで、真の原因に到達する。

## 5 Whys 分析

```
症状: アプリケーションがクラッシュ

なぜ1: メモリ不足エラーが発生
       ↓
なぜ2: 大量のオブジェクトが解放されていない
       ↓
なぜ3: イベントリスナーが削除されていない
       ↓
なぜ4: コンポーネントのクリーンアップ処理が未実装
       ↓
なぜ5: クリーンアップのベストプラクティスがドキュメント化されていない
       ↓
【根本原因】開発ガイドラインの不備
```

## 問題の定義（5W1H）

| 要素 | 質問 | 例 |
|------|------|-----|
| What | 何が起きたか | API呼び出しが500エラー |
| When | いつ起きたか | 本番デプロイ後 |
| Where | どこで起きたか | 経費申請機能 |
| Who | 誰が影響を受けたか | 全ユーザー |
| How | どのように起きたか | 特定条件で発生 |

## 原因の層別分析

```
症状
  └─> 直接原因（Immediate Cause）
      └─> 中間原因（Contributing Factors）
          └─> 根本原因（Root Cause）
              └─> システム的原因（Systemic Cause）
```

| 層 | 説明 | 対策レベル |
|---|------|---------|
| 直接原因 | 症状を引き起こした直接的な要因 | 応急処置 |
| 中間原因 | 直接原因が発生した背景 | 短期修正 |
| 根本原因 | 中間原因の根本にある問題 | 恒久対策 |
| システム的原因 | 組織やプロセスの問題 | 予防策 |

## Serena活用ポイント

```bash
# 1. エラーパターンの検索

# 2. 関連コードの分析
find_symbol("問題の関数/クラス", include_body=True)
find_referencing_symbols("問題箇所")

# 3. 類似問題の検索
read_memory("common_pitfalls_*")
```

## 対策マトリクス

| 対策 | 効果 | 実装コスト | 優先度 |
|------|------|-----------|--------|
| 短期（ワークアラウンド） | 低〜中 | 低 | 高 |
| 中期（修正） | 中〜高 | 中 | 中 |
| 長期（予防） | 高 | 高 | 計画的 |

## 使用タイミング

bug-fix-workflow の `phase-investigate.md` で基本調査を行い、以下の場合にこの詳細分析を実行：

- 再発する問題
- 影響範囲が広い問題
- 原因が複雑で特定困難な問題
- 設計上の問題が疑われる場合
