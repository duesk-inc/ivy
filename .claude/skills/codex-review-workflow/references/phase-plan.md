# Phase 2: 計画策定 (PLAN)

## 目的
調査結果を基に詳細設計と実装計画を策定する。
**計画完了後、自動的にCodex設計レビュー（Phase 3）に進む。**

## 実行手順

### 1. 既存パターン確認
- `.claude/rules/` 配下のルールファイルを参照
- `.claude/rules/05-pitfalls/known-issues.md` で既知の落とし穴を確認
- 類似機能の既存実装を調査

### 2. 設計項目
- データモデル設計
- API設計（エンドポイント、リクエスト/レスポンス）
- ビジネスロジック設計
- UI/UXコンポーネント設計
- セキュリティ設計

### 3. 実装計画（Phase分割）

**Phase A: データベース層**
- マイグレーション作成
- モデル定義
- シードデータ作成

**Phase B: リポジトリ層**
- インターフェース定義
- CRUD実装

**Phase C: サービス層**
- DTO定義
- ビジネスロジック実装

**Phase D: API層**
- ハンドラー作成
- ルート登録
- エラーハンドリング

**Phase E: フロントエンド**
- APIクライアント作成
- UIコンポーネント作成

**Phase F: テスト**
- ユニットテスト
- 統合テスト

### 4. 計画書の構成

以下の構成で計画書を作成する：

```markdown
# 実装計画: [機能名]

## 概要
[何を実装するか、なぜ必要か]

## 技術的アプローチ
[選択した技術方針とその理由]

## 変更対象ファイル
| ファイル | 変更内容 | 新規/修正 |
|---------|---------|----------|

## 実装順序
1. [ステップ1]
2. [ステップ2]

## データモデル変更（該当する場合）
[テーブル定義、マイグレーション]

## API設計（該当する場合）
[エンドポイント、リクエスト/レスポンス]

## 考慮事項
- セキュリティ
- パフォーマンス
- 後方互換性

## テスト方針
[テスト対象、テスト手法]
```

### 5. 出力
`docs/plan/feature-plan_{TIMESTAMP}.md`

## 終了条件
- SUCCESS → **自動的にCodex設計レビュー（Phase 3）へ進む**
- NEED_REDESIGN → Phase 1（INVESTIGATE）へ戻る
- NEED_CLARIFICATION → ユーザーに質問
