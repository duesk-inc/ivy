# Phase 2 設計判断ログ

日付: 2026-03-22

## 背景

Phase 1（1:1マッチング）は完成済み。Phase 2（Gmail連携 + N:Nマッチング）の実装計画を策定・Codexレビュー後、コスト・実用性を検討した結果、設計方針を大幅に見直した。

## 判断事項

### 1. N:Nマッチングは不要 → 凍結

**理由:**
- SES営業の実務は常に「案件起点」か「人材起点」。全組み合わせを一覧する場面がない
- コストが二乗で爆発する
  - 現在（500件/日）: 月$400
  - 将来（5,000件/日）: 月$41,000
- 1:Nマッチング（案件→人材、人材→案件）で実務は十分カバーできる

**対応:** コードは残すがUIから非表示にする。将来Embedding等で安くなった場合に復活の余地を残す。

### 2. サジェスト方式を採用

N:Nの代わりに「今日のマッチ候補」をダッシュボードに表示する方式。

**仕組み:**
1. メール同期時にルールベースで候補ペアを自動生成（コスト$0）
   - スキルキーワード重複、稼働時期一致、単価レンジ重複、条件一致
2. 上位20件を自動でClaude APIスコアリング（月$4程度）
3. 営業が朝アプリを開いた時に、スコア付き候補が表示済み

**コスト比較:**
| 方式 | 月額コスト |
|------|----------|
| N:Nバッチ（月8回） | $400〜$41,000 |
| サジェスト（20件/日自動） | ~$4 |

**ステータス:** 未実装。4月以降の営業フィードバック次第で着手。

### 3. 1:Nマッチング（案件→人材、人材→案件）を実装済み

- 案件一覧ページで「人材を探す」ボタン
- 人材一覧ページで「案件を探す」ボタン
- プレフィルタ後にClaude AIでスコアリング、結果をダイアログ表示
- コスト: 1回あたり~$0.14（プレフィルタ後20件程度）

### 4. 重複排除の方針

- 別会社から同じ案件が来ても「商流違い」として両方取り込む
- SHA256（送信者+件名+本文先頭500文字）で同一送信者からの重複のみ排除
- Phase 1のJobGroup機能で同一案件の複数経路を比較可能

### 5. Qoala vs Ivy

| 観点 | Qoala | Ivy |
|------|-------|-----|
| 月額 | 推定3.3万円〜（ユーザー数課金） | ~2万円（インフラ+API） |
| メール自動取込 | あり（実績あり） | Phase 2で実装済み（未運用） |
| 1:1詳細スコアリング | **なし** | **あり（7軸+NG判定+提案アドバイス）** |
| マッチング精度制御 | ブラックボックス | プロンプト調整可能 |
| Salesforce連携 | なし | Phase 3で予定 |
| スケール時コスト | ユーザー数で増加 | API従量のみ |
| 開発保守の手間 | ゼロ | 自社負担 |

**Ivyの差別化:** 1:1マッチングの詳細スコア（スキル・単価・時期・勤務地etc.の個別評価+提案アドバイス+確認ヒント）はQoalaにない機能。

**判断:** 4月入社の営業にPhase 1を使ってもらい、1:1スコアリングの実用性を検証。その結果でPhase 2をIvyで進めるかQoalaにするか決定。
